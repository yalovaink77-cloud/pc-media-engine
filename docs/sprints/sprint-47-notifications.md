# Sprint 47 — Notification Center

## Objective

Surface important platform events to operators through an in-app Notification Center. Notifications are **derived from existing audit events** — no workflow changes, no external delivery.

## Notification lifecycle

1. An operational action occurs (publish, queue pause, auth failure, etc.)
2. The action is recorded as an audit event (Sprint 46 — unchanged)
3. The notifying audit repository wrapper calls `NotificationService.notifyFromAudit()`
4. Matching audit event types are mapped to operator-facing notifications
5. Notifications are stored with `read: false` until marked read via API or dashboard
6. Operators view notifications in the dashboard Notification Center or via `GET /notifications`

Failures at step 3–4 are swallowed — notifications never block publishing, queue operations, or any other workflow.

## Notification model

| Field | Description |
|-------|-------------|
| `id` | UUID |
| `type` | Operator-facing type (e.g. `publish.failed`, `queue.paused`) |
| `category` | `publishing`, `queue`, `provider`, `system`, `security` |
| `severity` | `info`, `warn`, `error`, `critical` |
| `title` | Short headline |
| `message` | Human-readable detail |
| `read` | Boolean read/unread state |
| `correlationId` | From source audit event (`X-Request-Id`) |
| `auditEventId` | Link back to audit log entry |
| `metadata` | Copied from audit event |
| `createdAt` | ISO 8601 timestamp |

## Audit integration

Notifications are generated by `createNotifyingAuditRepository()`, a thin wrapper around `AuditRepository` that calls `notifyFromAudit()` after each successful `append()`. The `AuditService` interface is unchanged.

### Event mapping

| Audit event | Notification |
|-------------|--------------|
| `publishing.queued` | `publish.completed` |
| `publishing.failed` | `publish.failed` (or `queue.retry_exhausted` when retries exhausted) |
| `publishing.duplicate_skipped` | `publish.duplicate_skipped` |
| `queue.pause` | `queue.paused` |
| `queue.resume` | `queue.resumed` |
| `provider.health_check` (unhealthy) | `provider.unhealthy` |
| `provider.validation` / `provider.config_updated` (invalid) | `provider.config_invalid` |
| `system.startup` / `shutdown` / `fatal_error` | matching system notifications |
| `auth.rbac_denied` | `security.rbac_denied` |
| `auth.login_failure` (3+ in 5 min) | `security.auth_failures` |

## Read / unread

- New notifications default to `read: false`
- `POST /notifications/:id/read` — mark single notification read
- `POST /notifications/read-all` — mark all unread notifications read
- `GET /notifications?unread=true` — filter unread only
- Dashboard shows unread count badge on the Notifications nav link

## API

| Method | Path | Permission | Description |
|--------|------|------------|-------------|
| GET | `/notifications` | `notifications:read` | List with `unread`, `severity`, `limit` filters |
| POST | `/notifications/:id/read` | `notifications:read` | Mark one read |
| POST | `/notifications/read-all` | `notifications:read` | Mark all read |

RBAC: `notifications:read` granted to Admin, Operator, Publisher, and Viewer.

## Dashboard — Notification Center

Route: `GET /notifications`

Features:
- Bell indicator with unread badge in navigation
- Notification list with severity and read/unread badges
- Unread-only toggle
- Mark read / mark all read actions
- Notification detail with correlation ID and metadata JSON viewer

## Future external delivery

The `NotificationRepository` abstraction allows future sinks without changing audit or workflow code:
- Email / SMS / push notifications
- Slack or Teams webhooks
- PagerDuty incident creation

Sprint 47 stores notifications in-memory only — no external providers.

## Non-goals (Sprint 47)

- No email, SMS, push, Slack, or websocket delivery
